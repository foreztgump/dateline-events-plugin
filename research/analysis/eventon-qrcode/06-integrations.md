# eventon-qrcode 2.0.3 ‚Äî Integrations

---

## Integration Architecture

eventon-qrcode integrates with other systems in three directions: it hooks into EventON add-ons via WordPress actions/filters, it exposes a REST API for external scanner apps, and it calls an external HTTP service for QR image generation. All add-on integrations are guarded by the presence of the relevant hooks (if the RSVP or Tickets addon is inactive, those hooks simply never fire).

---

## 1. EventON Core (Required)

**Type:** Hard dependency.
**Check:** `plugins_loaded` callback ‚Äî aborts with admin notice if EventON core or the `evo_addons` class is absent.

The plugin reads all configuration from EventON's option store (`evcal_options_evcal_1`) via the `EVO()->cal` object. It contributes settings to EventON's admin tabs via `eventon_settings_tab1_arr_content`, `eventon_settings_lang_tab_content`, `eventon_appearance_add`, and `eventon_inline_styles_array`. The check-in page template resolution uses EventON's `eventon_template_paths` filter.

---

## 2. RSVP Addon (eventon-rsvp ‚Äî Optional)

**Type:** Optional integration; no `class_exists` guard ‚Äî coupling is purely hook-based.

| Hook | Direction | Purpose |
|---|---|---|
| `eventonrs_rsvp_post_table` | Inbound (listen) | Inject QR image row into admin RSVP list table |
| `evors_confirmation_email_before` | Inbound (listen) | Pre-generate and cache QR image before confirmation email sends |
| `eventonrs_confirmation_email` | Inbound (listen) | Append QR image block to RSVP confirmation email HTML |

The check-in page handles `evo-rsvp` post types natively in `evoqr_checkin::get_page_data()` ‚Äî it reads RSVP status via `EVO_RSVP_CPT`, enforces that the guest did not RSVP "no", and writes `status` postmeta (`check-in` / `checked`). Un-check (`?action=unc`) writes `check-in` directly via `update_post_meta`.

---

## 3. Tickets Addon (eventon-tickets / EVOTX ‚Äî Optional)

**Type:** Optional integration; hook-based.

| Hook | Direction | Purpose |
|---|---|---|
| `evotx_tixPost_tixid` | Filter (listen) | Hook point for ticket ID display ‚Äî returns value unchanged (passthrough in this version) |
| `evotx_email_tixid_list` | Filter (listen) | Replace ticket ID text in ticket emails with QR image + ticket number |
| `evotx_one_ticket_extra` | Action (listen) | Inject QR image on single ticket view (completed orders, non-refunded only) |

The check-in page handles `evo-tix` post types: it reads the ticket via `evotx_tix` and `EVO_Ticket`, checks WooCommerce order status (`WC_Order::get_status() == 'completed'`), and updates ticket status via `EVO_Ticket::set_status()`. Other tickets in the same WooCommerce order are displayed as sibling links.

---

## 4. WooCommerce (Indirect ‚Äî via Tickets Addon)

The plugin instantiates `WC_Order` directly (`new WC_Order($order_id)`) during the web check-in flow for ticket validation. This is an undeclared hard dependency when the Tickets addon is active ‚Äî if WooCommerce is absent but tickets exist in the database, the check-in page will throw a fatal error. The REST API check-in endpoint uses `EVO_Ticket::get_order_status()` instead, which may encapsulate this differently.

---

## 5. External QR Generation Service (api.qrserver.com)

**Type:** External HTTP dependency.
**Call site:** `evoqr_checkin::get_qr_code()` ‚Üí `get_uploaded_qr_code_image()`.

QR images are generated by calling:
```
GET https://api.qrserver.com/v1/create-qr-code/?data={url_encoded_checkin_url}&size={size}x{size}
```

The response (a PNG) is downloaded synchronously using WordPress's `download_url()`, then sideloaded into the WordPress media library via `media_handle_sideload()` into the `evo_qr_codes/` subdirectory with a 6-character random suffix appended to the filename. The resulting WP attachment URL is cached in postmeta (`_qrimg_{ticket_number}_{repeat_interval}`) so the external call is made only once per ticket.

**Failure handling:** If `download_url()` returns a `WP_Error`, the image upload is skipped and the function returns `false`, producing an empty `<img>` tag. There is no retry, no fallback generation, and no error logging.

**Privacy note:** The check-in URL (containing the encoded ticket number) is transmitted to a third-party service. The QR encoding is base64 only (see Security section below).

---

## 6. REST API (External Scanner App Integration)

**Type:** Outbound-facing API; opt-in via `evoqr_enable_api_access` setting.
**Namespace:** `eventon` (WP REST API).

All four endpoints are registered under `/wp-json/eventon/`:

| Method | Route | Purpose |
|---|---|---|
| GET | `/eventon/events_list` | List all published events with attendee type (`rsvp` / `tickets` / `no_attendees`) and basic metadata |
| GET | `/eventon/attendees_list/{event_uid}` | All RSVP guests and/or ticket holders for an event; `event_uid` format is `{post_id}_{repeat_interval}` |
| GET | `/eventon/one_attendee/{ticket_number}` | Single attendee record with name, email, check status, order info, and event metadata |
| GET | `/eventon/checkin/{ticket_number}?checkin_status=checked\|check-in` | Set check-in status for a ticket or RSVP; supports both check-in and un-check in one endpoint |

All endpoints return `WP_REST_Response` with status code 201 and a `Location: http://myeventon.com/` header (appears to be a development artefact ‚Äî not meaningful).

**`/events_list`** passes EventON's standard event query and returns per-event: name, start timestamp (Unix), raw start, duration, timezone, location name, and `event_type`. For repeating events, each occurrence is listed separately with its unique ID (`{post_id}_{repeat_interval}`).

**`/attendees_list/{event_uid}`** merges RSVP guests (via `EVORS_Event::GET_rsvp_list()`) and ticket holders (via `EVOTX_Attendees::get_tickets_for_event()`), keyed by attendee/ticket ID. Fields returned: `attendee_id`, `ticket_number`, `first_name`, `last_name`, `check_status`, `rsvp_status` (RSVP only), `email`, `count`, `order_id`, `order_status` (tickets only).

**`/one_attendee/{ticket_number}`** returns the same fields plus event name, start, and duration.

**`/checkin/{ticket_number}`** validates the current status, checks WooCommerce order completion for tickets, checks RSVP "no" status for RSVPs, and writes the new status. Returns a `message` / `message_code` pair in the response body.

### Critical Security Issue: Unauthenticated REST Endpoints

**All four endpoints use `'permission_callback' => function() { return true; }`** ‚Äî no authentication whatsoever. The commented-out alternative (`is_user_have_permission_to_checkin()`) was never activated. This means:

- Any unauthenticated HTTP client can list all events with attendee types
- Any client can retrieve all attendee names, emails, and check-in statuses for any event
- Any client can check in or un-check any attendee without credentials

This is a high-severity vulnerability: full attendee PII (name, email) is publicly readable and the check-in state mutation requires no auth. The only protection is that the API must be explicitly enabled via the admin setting (`evoqr_enable_api_access`), but once enabled it is entirely open.

## 7. WordPress Media Library (Internal)

QR images are stored as standard WP attachment posts via `media_handle_sideload()`. The plugin redirects the upload path to `{uploads_dir}/evo_qr_codes/` using the `upload_dir` filter, gated on a `$_POST['type'] == 'eventon_qr_code'` flag set during the sideload call. The `wp_unique_filename` filter appends a 6-character random alphanumeric suffix to each filename to prevent collisions. The `posts_where` filter excludes QR images from the media library UI unless `evoqr_show_in_media` is enabled.

## Dateline Design Notes

1. **QR generation**: Replace api.qrserver.com with server-side generation (e.g., `qrcode` npm package) to eliminate the third-party HTTP call, PII leakage, and synchronous failure risk. Generate at RSVP/ticket creation time via `ctx.waitUntil`; cache the result URL in KV (`qr:{ticket_id}`, TTL: event date + 30 days).
2. **Token security**: Use HMAC-SHA256 signed tokens (`ticket_id:event_id:timestamp`, short TTL) instead of base64. Consider one-time-use tokens for high-security events.
3. **REST API authentication**: All check-in endpoints must require a `write:checkin` capability (staff role). The unauthenticated pattern in the original is a critical flaw to not replicate.
4. **Attendee list API**: Needs pagination (the original returns unbounded result sets) and authentication before exposing any attendee PII.
5. **WooCommerce coupling**: Dateline has no WooCommerce; ticket check-in validation should check the Dateline order/payment status in KV or `ctx.content` rather than instantiating an ORM object.
